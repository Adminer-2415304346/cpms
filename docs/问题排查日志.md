## 二、MySQL 数据库

### 问题 1：root 无密码连接被拒

**现象**：
```
ERROR 1045 (28000): Access denied for user 'root'@'localhost' (using password: NO)
```

**原因**：MySQL 安装时设置了 root 密码 `123456`，命令行未提供。

**解决**：所有 mysql 命令加 `-p123456` 参数。

```bash
mysql -u root -p123456
```

---

### 问题 2：视图创建报语法错误 —— `or` 是保留字

**现象**：
```
ERROR 1064 (42000): You have an error in your SQL syntax near 'or ON r.room_id = or.room_id'
```

**原因**：在视图 `v_overdue_bills` 中使用了 `or` 作为 `owner_room` 表的别名，而 `OR` 是 MySQL 保留字。

**错误 SQL**：
```sql
JOIN owner_room or ON r.room_id = or.room_id
JOIN owner o       ON or.owner_id = o.owner_id
```

**修复**：将别名 `or` 改为 `owr`：
```sql
JOIN owner_room owr ON r.room_id = owr.room_id
JOIN owner o        ON owr.owner_id = o.owner_id
```

**教训**：避免使用 SQL 保留字作为别名（`or`、`and`、`order`、`group` 等）。

---

### 问题 3：中文数据插入报 "Data too long"

**现象**：
```
ERROR 1406 (22001): Data too long for column 'real_name' at row 1
```

**原因**：Windows 命令行默认字符集与 MySQL 的 utf8mb4 不匹配，导致中文字符被错误编码。

**解决**：连接时显式指定字符集：
```bash
mysql -u root -p123456 --default-character-set=utf8mb4 < schema.sql
```

---

### 问题 4：不能在同一条 `-e` 命令中混合 SQL 和 SOURCE

**现象**：
```
ERROR 1064: You have an error... near 'SOURCE E:/cpms/docs/schema.sql' at line 1
```

**原因**：`SOURCE` 是 mysql 客户端命令，不能写在 `-e` 的 SQL 字符串中。

**错误写法**：
```bash
mysql -u root -p123456 -e "DROP DATABASE IF EXISTS cpms; SOURCE schema.sql"
```

**正确写法**（分两步或使用管道）：
```bash
mysql -u root -p123456 -e "DROP DATABASE IF EXISTS cpms"
mysql -u root -p123456 < schema.sql
```

### 问题 5：JPA 校验报列类型不匹配

**现象**：
```
Schema-validation: wrong column type encountered in column [status] in table [bill];
found [enum (Types#CHAR)], but expecting [varchar(10) (Types#VARCHAR)]
```

**原因**：`schema.sql` 中使用了 MySQL 原生 `ENUM(...)` 类型，但 JPA 实体中映射的是 `String` + `@Column(length=10)`（即 VARCHAR）。Hibernate 的 `ddl-auto=validate` 模式在启动时发现两者不匹配，拒绝启动。

**修复**：将 `application.properties` 中的校验模式改为 `none`（因为表已由 SQL 脚本手动创建，无需 Hibernate 校验）：

```properties
# 修改前
spring.jpa.hibernate.ddl-auto=validate

# 修改后
spring.jpa.hibernate.ddl-auto=none
```

| `ddl-auto` 值 | 含义 | 适用场景 |
|:--:|------|------|
| `validate` | 校验实体与表结构一致 | 生产环境 |
| `none` | 不做任何操作 | 手动管理表结构 |
| `update` | 自动修改表结构 | 开发环境 |
| `create` | 每次启动重建表 | 测试环境 |

## 三、Spring Boot 后端

### 问题 6：Maven 不在 PATH 中

**现象**：
```
mvn: command not found
```

**原因**：Maven 安装在 `D:\apache-maven-3.9.16-bin\apache-maven-3.9.16\`，但未加入系统环境变量 PATH。

**解决**：使用完整路径调用 Maven：
```bash
D:\apache-maven-3.9.16-bin\apache-maven-3.9.16\bin\mvn <goal>
```

**建议**：将 Maven bin 目录加入系统 PATH，一劳永逸。

---

### 问题 7：JDK 版本与 Spring Boot 版本不匹配

**现象**：本机安装的是 **JDK 25**，但 `pom.xml` 配置的是 Java 17 + Spring Boot 3.2.5。

**原因**：Spring Boot 3.2.x 最高支持 Java 21，无法编译 Java 25。

**修复**：升级到 Spring Boot 3.5.0（实测可用）：

```xml
<!-- 修改前 -->
<parent>
    <version>3.2.5</version>
</parent>
<java.version>17</java.version>

<!-- 修改后 -->
<parent>
    <version>3.5.0</version>
</parent>
<java.version>25</java.version>
```

Spring Boot 3.5.0 对应 Spring Framework 6.2.7，Hibernate 6.6.15.Final。

### 问题 8：Lombok 注解不生效 —— getter/setter 未生成

**现象**：编译报大量 "找不到符号" 错误，全是 `getName()`、`setName()`、`getStatus()` 等方法。

```
[ERROR] 找不到符号
  符号:   方法 getType()
  位置: 类型为com.cpms.entity.ParkingSpot的变量 updated
```

**原因**：JDK 25 修改了 `sun.misc.Unsafe` 相关 API，低版本 Lombok 的注解处理器无法在高版本 JDK 下正常工作。

**修复**：
1. 显式指定 Lombok 版本为 `1.18.38`（支持 JDK 25）
2. 在 `pom.xml` 中增加 `maven-compiler-plugin` 显式配置注解处理器路径：

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <source>25</source>
        <target>25</target>
        <annotationProcessorPaths>
            <path>
                <groupId>org.projectlombok</groupId>
                <artifactId>lombok</artifactId>
                <version>1.18.38</version>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>
```

编译后的警告可忽略：
```
WARNING: sun.misc.Unsafe::objectFieldOffset has been called by lombok.permit.Permit
WARNING: sun.misc.Unsafe::objectFieldOffset will be removed in a future release
```

### 问题 9：Spring Data JPA 方法名解析错误

**现象**：
```
No property 'id' found for type 'User'; Traversed path: Owner.user
```

**原因**：`OwnerRepository` 中定义了方法 `findByUserId(Long userId)`。Spring Data JPA 按驼峰命名规则将其解析为 `Owner.user.id`（从 Owner → User → id）。但 User 实体的主键字段是 `userId` 而非 `id`。

**Spring Data JPA 方法名解析规则**：

| 方法名 | 解析结果 |
|--------|----------|
| `findByUserId` | `owner.user.id` ❌ (User 没有 `id` 字段) |
| `findByUser_UserId` | `owner.user.userId` ✅ (用下划线分隔嵌套路径) |

**修复**：将方法名改为 `findByUser_UserId`，同时更新 Service 层调用。

```java
// 修改前
Optional<Owner> findByUserId(Long userId);

// 修改后
Optional<Owner> findByUser_UserId(Long userId);
```

**教训**：当实体使用非标准主键名（如 `userId` 而非 `id`）时，Spring Data JPA 的 Repository 方法名必须使用下划线 `_` 来明确嵌套属性边界。

### 问题 10：登录密码不匹配

**现象**：用户在前端输入 `123456` 点击登录，返回 "用户名或密码错误"。

**原因**：`schema.sql` 示例数据中的密码存的是 MD5 哈希值 `e10adc3949ba59abbe56e057f20f883e`（即 `123456` 的 MD5），但后端 `UserService.login()` 中直接比较了用户输入的明文 `123456` 与数据库中的哈希值，两者不匹配。

```java
// UserService.java 中的登录逻辑（明文比较）
if (!user.getPassword().equals(req.getPassword())) {
    throw new RuntimeException("用户名或密码错误");
}
```

**解决**：将数据库中所有用户密码重置为明文 `123456`：
```sql
UPDATE user SET password = '123456';
```

**注意**：课程设计项目为简化处理使用明文密码。正式项目应对密码加盐哈希存储，并在登录时对输入做同样哈希后比较。

## 四、Vue 前端

### 问题 11：登录页风格陈旧

**现象**：初版登录页使用深蓝单色背景 + 静态圆环装饰，缺少动感和现代感。

**解决**：改用紫蓝渐变 + 5 个浮动光斑（CSS keyframes 位移动画）+ Canvas 粒子网络（40 个粒子 + 近距离连线）+ 卡片入场动画 + 按钮光泽流转效果。

**关键 CSS 技术**：
- `filter: blur(80px)` 实现光斑柔和感
- `@keyframes` 实现不同周期/方向的浮动
- `requestAnimationFrame` Canvas 粒子系统
- `backdrop-filter: blur(24px)` 玻璃态效果
