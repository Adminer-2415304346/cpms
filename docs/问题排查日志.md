## 二、MySQL 数据库

### 问题 1：root 无密码连接被拒

**现象**：
```
ERROR 1045 (28000): Access denied for user 'root'@'localhost' (using password: NO)
```

**原因**：MySQL 安装时设置了 root 密码 `123456`，命令行未提供。

**解决**：所有 mysql 命令加 `-p123456` 参数。

```bash
mysql -u root -p123456
```

---

### 问题 2：视图创建报语法错误 —— `or` 是保留字

**现象**：
```
ERROR 1064 (42000): You have an error in your SQL syntax near 'or ON r.room_id = or.room_id'
```

**原因**：在视图 `v_overdue_bills` 中使用了 `or` 作为 `owner_room` 表的别名，而 `OR` 是 MySQL 保留字。

**错误 SQL**：
```sql
JOIN owner_room or ON r.room_id = or.room_id
JOIN owner o       ON or.owner_id = o.owner_id
```

**修复**：将别名 `or` 改为 `owr`：
```sql
JOIN owner_room owr ON r.room_id = owr.room_id
JOIN owner o        ON owr.owner_id = o.owner_id
```

**教训**：避免使用 SQL 保留字作为别名（`or`、`and`、`order`、`group` 等）。

---

### 问题 3：中文数据插入报 "Data too long"

**现象**：
```
ERROR 1406 (22001): Data too long for column 'real_name' at row 1
```

**原因**：Windows 命令行默认字符集与 MySQL 的 utf8mb4 不匹配，导致中文字符被错误编码。

**解决**：连接时显式指定字符集：
```bash
mysql -u root -p123456 --default-character-set=utf8mb4 < schema.sql
```

---

### 问题 4：不能在同一条 `-e` 命令中混合 SQL 和 SOURCE

**现象**：
```
ERROR 1064: You have an error... near 'SOURCE E:/cpms/docs/schema.sql' at line 1
```

**原因**：`SOURCE` 是 mysql 客户端命令，不能写在 `-e` 的 SQL 字符串中。

**错误写法**：
```bash
mysql -u root -p123456 -e "DROP DATABASE IF EXISTS cpms; SOURCE schema.sql"
```

**正确写法**（分两步或使用管道）：
```bash
mysql -u root -p123456 -e "DROP DATABASE IF EXISTS cpms"
mysql -u root -p123456 < schema.sql
```

### 问题 5：JPA 校验报列类型不匹配

**现象**：
```
Schema-validation: wrong column type encountered in column [status] in table [bill];
found [enum (Types#CHAR)], but expecting [varchar(10) (Types#VARCHAR)]
```

**原因**：`schema.sql` 中使用了 MySQL 原生 `ENUM(...)` 类型，但 JPA 实体中映射的是 `String` + `@Column(length=10)`（即 VARCHAR）。Hibernate 的 `ddl-auto=validate` 模式在启动时发现两者不匹配，拒绝启动。

**修复**：将 `application.properties` 中的校验模式改为 `none`（因为表已由 SQL 脚本手动创建，无需 Hibernate 校验）：

```properties
# 修改前
spring.jpa.hibernate.ddl-auto=validate

# 修改后
spring.jpa.hibernate.ddl-auto=none
```

| `ddl-auto` 值 | 含义 | 适用场景 |
|:--:|------|------|
| `validate` | 校验实体与表结构一致 | 生产环境 |
| `none` | 不做任何操作 | 手动管理表结构 |
| `update` | 自动修改表结构 | 开发环境 |
| `create` | 每次启动重建表 | 测试环境 |

## 三、Spring Boot 后端

### 问题 6：Maven 不在 PATH 中

**现象**：
```
mvn: command not found
```

**原因**：Maven 安装在 `D:\apache-maven-3.9.16-bin\apache-maven-3.9.16\`，但未加入系统环境变量 PATH。


### 问题 7：JDK 版本与 Spring Boot 版本不匹配

**现象**：本机安装的是 **JDK 25**，但 `pom.xml` 配置的是 Java 17 + Spring Boot 3.2.5。

**原因**：Spring Boot 3.2.x 最高支持 Java 21，无法编译 Java 25。

**修复**：升级到 Spring Boot 3.5.0（实测可用）：

```xml
<!-- 修改前 -->
<parent>
    <version>3.2.5</version>
</parent>
<java.version>17</java.version>

<!-- 修改后 -->
<parent>
    <version>3.5.0</version>
</parent>
<java.version>25</java.version>
```

Spring Boot 3.5.0 对应 Spring Framework 6.2.7，Hibernate 6.6.15.Final。

### 问题 8：Lombok 注解不生效 —— getter/setter 未生成

**现象**：编译报大量 "找不到符号" 错误，全是 `getName()`、`setName()`、`getStatus()` 等方法。

```
[ERROR] 找不到符号
  符号:   方法 getType()
  位置: 类型为com.cpms.entity.ParkingSpot的变量 updated
```

**原因**：JDK 25 修改了 `sun.misc.Unsafe` 相关 API，低版本 Lombok 的注解处理器无法在高版本 JDK 下正常工作。

**修复**：
1. 显式指定 Lombok 版本为 `1.18.38`（支持 JDK 25）
2. 在 `pom.xml` 中增加 `maven-compiler-plugin` 显式配置注解处理器路径：

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <source>25</source>
        <target>25</target>
        <annotationProcessorPaths>
            <path>
                <groupId>org.projectlombok</groupId>
                <artifactId>lombok</artifactId>
                <version>1.18.38</version>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>
```

编译后的警告可忽略：
```
WARNING: sun.misc.Unsafe::objectFieldOffset has been called by lombok.permit.Permit
WARNING: sun.misc.Unsafe::objectFieldOffset will be removed in a future release
```

### 问题 9：Spring Data JPA 方法名解析错误

**现象**：
```
No property 'id' found for type 'User'; Traversed path: Owner.user
```

**原因**：`OwnerRepository` 中定义了方法 `findByUserId(Long userId)`。Spring Data JPA 按驼峰命名规则将其解析为 `Owner.user.id`（从 Owner → User → id）。但 User 实体的主键字段是 `userId` 而非 `id`。

**Spring Data JPA 方法名解析规则**：

| 方法名 | 解析结果 |
|--------|----------|
| `findByUserId` | `owner.user.id` ❌ (User 没有 `id` 字段) |
| `findByUser_UserId` | `owner.user.userId` ✅ (用下划线分隔嵌套路径) |

**修复**：将方法名改为 `findByUser_UserId`，同时更新 Service 层调用。

```java
// 修改前
Optional<Owner> findByUserId(Long userId);

// 修改后
Optional<Owner> findByUser_UserId(Long userId);
```

**教训**：当实体使用非标准主键名（如 `userId` 而非 `id`）时，Spring Data JPA 的 Repository 方法名必须使用下划线 `_` 来明确嵌套属性边界。

### 问题 10：登录密码不匹配

**现象**：用户在前端输入 `123456` 点击登录，返回 "用户名或密码错误"。

**原因**：`schema.sql` 示例数据中的密码存的是 MD5 哈希值 `e10adc3949ba59abbe56e057f20f883e`（即 `123456` 的 MD5），但后端 `UserService.login()` 中直接比较了用户输入的明文 `123456` 与数据库中的哈希值，两者不匹配。

```java
// UserService.java 中的登录逻辑（明文比较）
if (!user.getPassword().equals(req.getPassword())) {
    throw new RuntimeException("用户名或密码错误");
}
```

**解决**：将数据库中所有用户密码重置为明文 `123456`：
```sql
UPDATE user SET password = '123456';
```

**注意**：课程设计项目为简化处理使用明文密码。正式项目应对密码加盐哈希存储，并在登录时对输入做同样哈希后比较。

### 问题 12：日期序列化格式不匹配（致命）

**现象**：
- 创建账单 `POST /api/bills` 报 400
- 创建/编辑业主 `POST/PUT /api/owners` 报 400
- 浏览器 Network 面板显示请求状态码 400 Bad Request

**原因**：前端 Ant Design Vue 的 `<a-date-picker>` 组件 v-model 绑定的是 Day.js 对象。Axios 序列化 JSON 时，Day.js 的 `.toJSON()` 输出 ISO 日期时间字符串（如 `"2026-06-30T00:00:00+08:00"`），但后端 `Bill.dueDate` 和 `Owner.moveInDate` 字段类型是 `LocalDate`。Jackson 默认的 `LocalDateDeserializer` 只能解析纯日期格式 `"yyyy-MM-dd"`，遇到带时间部分的字符串直接抛 `DateTimeParseException`，最终被全局异常处理器转为 400。

**涉及字段**：
| 实体 | 字段 | 类型 | 前端来源 |
|------|------|------|----------|
| Bill | dueDate | LocalDate | `<a-date-picker>` |
| Owner | moveInDate | LocalDate | `<a-date-picker>` |

**修复**：新建 `JacksonConfig.java`，注册自定义 `LocalDate` 反序列化器，按优先级依次尝试四种格式：

```
1. ISO_LOCAL_DATE       →  "2026-06-30"
2. ISO_OFFSET_DATE_TIME →  "2026-06-30T00:00:00+08:00"
3. ISO_OFFSET_DATE_TIME →  "2026-06-30T00:00:00.000Z" (先转换 Z → +00:00)
4. ISO_LOCAL_DATE_TIME  →  "2026-06-30T00:00:00"
```

```java
@Configuration
public class JacksonConfig {
    @Bean
    public ObjectMapper objectMapper(Jackson2ObjectMapperBuilder builder) {
        JavaTimeModule module = new JavaTimeModule();
        module.addDeserializer(LocalDate.class, new JsonDeserializer<LocalDate>() {
            // 依次用四种格式尝试解析
        });
        return builder.modules(module).build();
    }
}
```

### 问题 13：全局异常处理器把一切异常都返回 400

**现象**：浏览器控制台无论什么错误都显示 400，无法区分"请求参数有误"和"服务器挂了"。

**原因**：`GlobalExceptionHandler` 中 `@ExceptionHandler(RuntimeException.class)` 的 `@ResponseStatus` 写的是 `HttpStatus.BAD_REQUEST`（400），但 Service 层大量使用 `throw new RuntimeException("xxx不存在")`。这导致资源未找到（应为 404）、业务校验失败（应为 400）、内部未知错误（应为 500）全部以 400 返回。

**旧代码**：
```java
@ExceptionHandler(RuntimeException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)   // ← 所有异常一律 400
public ApiResult<Void> handleRuntime(RuntimeException e) {
    return ApiResult.error(e.getMessage());
}
```

**修复**：
1. 新建 `NotFoundException` 异常类（继承 RuntimeException）
2. 将 Service 层所有 "xxx不存在" 的异常从 `RuntimeException` 改为 `NotFoundException`（涉及 9 个 Service，共 11 处）
3. 重写 `GlobalExceptionHandler`，按异常类型分层：

| 异常类型 | HTTP 状态码 | 含义 |
|----------|:----------:|------|
| `NotFoundException` | 404 | 资源未找到 |
| `MethodArgumentNotValidException` | 400 | 参数校验失败 |
| `HttpMessageNotReadableException` | 400 | JSON 解析失败（含日期格式错误） |
| `IllegalArgumentException` | 400 | 业务校验错误 |
| `RuntimeException` | 500 | 其他未预期错误 |

### 问题 14：Hibernate 懒加载代理序列化隐患

**现象**：部分列表接口偶发报错，错误信息含 `LazyInitializationException` 或 `could not initialize proxy`。

**原因**：JPA 实体间存在 `@ManyToOne(fetch = FetchType.LAZY)` 关系（如 Room → Building、Bill → Room、Repair → Owner 等）。Jackson 序列化实体时，如果 Hibernate Session 恰好已关闭（Open Session In View 超时或关闭），访问未初始化的懒加载代理就会抛 `LazyInitializationException`。此外，Hibernate 生成的代理类包含 `hibernateLazyInitializer` 和 `handler` 两个额外属性，Jackson 可能尝试序列化它们导致异常。

**修复**：
1. 给全部 12 个实体类加上 `@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})`，让 Jackson 忽略 Hibernate 代理的内部字段
2. 在 `application.properties` 中显式保持 OSIV 开启（Spring Boot 默认开启，此处仅做防御）

```java
@Data
@Entity
@Table(name = "user")
@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})
public class User { ... }
```

### 问题 15：API 响应泄露用户密码

**现象**：调用 `GET /api/users` 或 `GET /api/owners` 时，返回的 JSON 中包含 `password` 字段，前端可直接看到所有用户的密码明文。

**原因**：`User` 实体使用 `@Data` 注解，Lombok 为所有字段生成了 getter/setter，Jackson 序列化时将所有字段都输出到 JSON。`password` 字段没有做任何遮蔽处理。

**修复**：给 `User.password` 字段加上 `@JsonProperty(access = JsonProperty.Access.WRITE_ONLY)`：

```java
@JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
@Column(nullable = false, length = 255)
private String password;
```

效果：前端发送的 JSON 中的 `password` 可以被正常接收（反序列化），但后端返回的 JSON 中不包含 `password` 字段（序列化时跳过）。

### 问题 16：报修列表——业主查询用错 ID

**现象**：业主登录后，报修列表为空（明明自己提交过报修），控制台无报错只是返回空数组。

**原因**：前端 `RepairList.vue` 的 `fetch()` 方法中，业主角色直接使用 `store.userId`（用户表主键）作为查询参数 `ownerId`（业主表主键）：

```javascript
// 错误：store.userId 是 user 表的 ID，不是 owner 表的 ID
else if (isOwner.value) url += `?ownerId=${store.userId}`
```

用户表和业主表是两张独立的表，ID 通常不一致。例如：用户 ID 为 3（user 表），但对应业主记录 ID 为 1（owner 表）。用 userId=3 去查 owner_id=3 的报修工单，自然查不到。

**修复**：先通过 userId 查出 owner，再用 owner.ownerId 去查报修：

```javascript
else if (isOwner.value) {
    const owner = await http.get(`/owners/by-user/${store.userId}`)
    if (owner) url += `?ownerId=${owner.ownerId}`
}
```

### 问题 17：投诉建议——业主未过滤，看到全部数据

**现象**：业主登录后进入投诉建议页面，看到的是所有人的投诉/建议，而非仅自己的。

**原因**：`ComplaintList.vue` 的 `fetch()` 方法完全没有对业主角色做 `ownerId` 过滤，查询参数只有 `?status=X` 可选筛选。而 `RepairList.vue` 至少尝试了过滤（虽然 ID 用错了）。

**修复**：加入与报修列表一致的过滤逻辑——业主角色时，先查 owner 再按 `ownerId` 过滤：

```javascript
} else if (isOwner.value) {
    const owner = await http.get(`/owners/by-user/${store.userId}`)
    if (owner) url += `?ownerId=${owner.ownerId}`
}
```

### 问题 18：缴费后账单状态未更新（可重复支付）

**现象**：对一笔"未缴"账单执行缴费操作，API 返回成功，但账单列表中该账单仍显示"未缴"，且"缴费"按钮仍然可见，可以重复支付同一笔账单。

**原因**：`BillService.pay()` 方法只创建并保存了 `Payment` 实体，但**没有更新 `Bill.status` 为 `"paid"`**。虽然代码开头有 `if ("paid".equals(bill.getStatus()))` 校验，但首次缴费后 status 未变，第二次调用时 status 仍是 `"unpaid"`，校验直接通过，允许重复支付。

**旧代码**：
```java
@Transactional
public Payment pay(Long billId, String payMethod, Long staffId) {
    Bill bill = getById(billId);
    if ("paid".equals(bill.getStatus())) {
        throw new RuntimeException("该账单已缴费");
    }
    Payment payment = new Payment();
    // ... 设置 payment 各字段
    return paymentRepository.save(payment);
    // ← 缺少 bill.setStatus("paid")！
}
```

**修复**：在校验通过后、创建 Payment 之前，标记账单状态：

```java
bill.setStatus("paid");   // ← 加这一行
Payment payment = new Payment();
```

由于方法上有 `@Transactional`，JPA 脏检查会自动将 Bill 状态变更同步到数据库。
